# SupplyMind: Complete Project Documentation (v2.0)

Welcome to the comprehensive, up-to-date documentation for **SupplyMind** — an Agentic AI Operating System for Supply Chain Intelligence. ## 1. High-Level System Architecture

SupplyMind uses a **client-server** architecture with a **Multi-Agent AI Layer** powered by LangChain + LangGraph, backed by a local Ollama LLM (llama3.2:latest) with smart mock fallback. **One module — Supplier Risk Scoring — uses a trained classical ML model (`backend/ml/pipeline.pkl`) instead of the LLM**, enabling deterministic, fast, explainable risk predictions without any AI inference call.

### 2.3 Agent State System

The backend holds a global in-memory dict `agent_states` keyed by `AgentRole` enum:

```python
AgentRole: ORCHESTRATOR | DEMAND | INVENTORY | SUPPLIER | ACTION | LOGISTICS | PLANNING | SUSTAINABILITY
AgentStatus: IDLE | THINKING | ACTING | WAITING
```

`update_agent_state(agent, status, task)` is called at the start/end of every AI operation. The frontend polls `/api/agents/states` every **3 seconds** to reflect live status in the sidebar indicator dots.

An `agent_messages` ring buffer (max 100) stores inter-agent communication events (task delegations, tool calls, analysis completions, synthesis events). These are exposed at `/api/agents/messages`.

### 2.4 Priority System

A `PRIORITY_ORDER = {"critical": 0, "high": 1, "medium": 2, "low": 3}` dict drives `sort_by_priority()` and `needs_attention()` helpers used in every workflow. All workflow outputs are priority-ranked before being returned to the frontend.

## 3. Backend — `backend/langchain_agents.py` (1114 lines)

This is the brain of SupplyMind. It contains all 12 tools, the 8 agent prompts, the `SupplyChainAgentSystem` class, and the 4-step `business_query()` pipeline.

### 3.1 The 12 LangChain Tools (@tool decorated)

| Tool Name | Agent | Core Data Source | What It Does |
|---|---|---|---|
| `forecast_demand` | Demand | `DEMAND_DATA` | Historical avg + trend % + MAPE + 3-period forecasts with 95% confidence bands |
| `analyze_stockout_risk` | Demand | `INVENTORY_DATA`` | Compares days_of_supply vs lead_time; assigns critical/high/medium/low risk per warehouse |
| `calculate_reorder_point` | Inventory | `DEMAND_DATA`, `PRODUCTS` | EOQ + safety stock (z-score based) + reorder point |
| `check_warehouse_levels` | Inventory | `INVENTORY_DATA` | Critical/Low/Optimal/Overstock SKU counts + health score |
| `score_supplier_risk` | Supplier | **`backend/ml/pipeline.pkl`** (ML model) + `get_supplier_risk_data()` | **ML-powered**: loads the trained scikit-learn pipeline from `backend/ml/pipeline.pkl`, engineers features from supplier data (reliability_score, on_time_delivery, quality_rating, financial_health, negative_news_count, shipment delay rate), runs `pipeline.predict_proba()`, applies the saved `threshold.pkl` to output a 0–100 risk score and critical/high/medium/low label. No LLM is called. |
| `get_shipping_options` | Supplier | Hardcoded modes per region | Selects Air/Sea/Rail/Ground based on urgency (urgent/normal/economy) |
| `get_market_intelligence` | Supplier | `NEWS_EVENTS` | Filters news by topic keywords; sentiment analysis |
| `predict_shipment_delay` | Supplier | `SHIPMENT_DATA`, `NEWS_EVENTS` | Historical delay rate + news bump → adjusted probability % |
| `generate_purchase_order` | Action | `PRODUCTS`, `SUPPLIERS` | Drafts PO with PO number, cost, expected delivery |
| `optimize_logistics_cost` | Logistics | `COST_DATA` | Category totals over N months; identifies Transportation/Warehousing/Inventory savings |
| `generate_production_plan` | Planning | `DEMAND_DATA`, `INVENTORY_DATA` | 8-week schedule with seasonality factors; flags critical/overstock weeks |
| `track_sustainability` | Sustainability | `SHIPMENT_DATA`, `NEWS_EVENTS`, `SUPPLIERS` | CO₂ estimation by carrier mode; ESG news; green action plan |

### 3.2 Tool Registry

```python
DEMAND_TOOLS       = [forecast_demand, analyze_stockout_risk]
INVENTORY_TOOLS    = [calculate_reorder_point, check_warehouse_levels]
SUPPLIER_TOOLS     = [score_supplier_risk, get_shipping_options, get_market_intelligence, predict_shipment_delay]
ACTION_TOOLS       = [generate_purchase_order]
LOGISTICS_TOOLS    = [optimize_logistics_cost]
PLANNING_TOOLS     = [generate_production_plan]
SUSTAINABILITY_TOOLS = [track_sustainability]

ALL_TOOLS          = (all of the above = 12 tools)
TOOL_MAP           = {t.name: t for t in ALL_TOOLS}   # fast O(1) lookup
AGENT_TOOL_MAP     = {"demand": DEMAND_TOOLS, ...}
```

### 3.3 The `SupplyChainAgentSystem` Class

#### Constructor
- Attempts to connect `ChatOllama(model="llama3.2:latest")` at `http://localhost:11434`
- If Ollama unavailable → `self.llm = None` (all agents run in mock mode)
- 8 agent system prompts created in `_create_agent_prompts()`; each returns structured JSON

#### Key Methods

**`invoke_tool(tool_name, **kwargs)`** — Calls the tool function via `TOOL_MAP[name].invoke(kwargs)`, parses JSON with a custom `NumpyEncoder` to handle numpy types cleanly.

**`run_agent(agent_name, task, context)`** — Formats a `SystemMessage` + `HumanMessage` and calls `llm.ainvoke()`. Falls back to `_mock_agent_response()` if LLM unavailable or fails. All responses are expected as JSON `{analysis, recommendations, confidence}`.

**`_mock_agent_response(agent_name, task, context)`** — Context-aware mock: reads real context keys (`critical_items_count`, `high_risk_count`, etc.) to produce meaningful (but deterministic) responses.

**`business_query(query)`** — The primary intelligence pipeline (4 steps):

```
Step 1: Orchestrator Agent (LLM) reads query → decides which agents to activate
         Returns: {agents_involved: [...], reasoning: "...", urgency: "..."}
         Fallback: keyword-matching on query text
Step 2: For each activated agent, call its real tools over all products/suppliers:
         - demand   → forecast_demand + analyze_stockout_risk  (all 10 products)
         - inventory→ calculate_reorder_point (all 10 products)
         - supplier → score_supplier_risk + predict_shipment_delay (all 8 suppliers)
                      NOTE: score_supplier_risk uses ML model (pipeline.pkl), NOT LLM
         - logistics→ optimize_logistics_cost (3-month window)
         - planning → generate_production_plan (top 3 products)
         - sustainability → track_sustainability (Global)
Step 3: Each activated agent's LLM (or mock) reasons over its real tool data
         NOTE: Risk scores fed into this step are already ML-computed numerics,
               so the LLM only adds narrative interpretation, not calculation
Step 4: Synthesise into:
         - priority_actions[] (sorted critical→low, max 10)
         - seven_day_plan[] (Day1-2, Day3-4, Day5-6, Day7)
         - executive_summary (LLM or data-driven mock)
         - key_metrics {critical_items, high_risk_suppliers, logistics_savings, ...}
         - agent_insights {agent: {analysis, confidence}}
```

**`orchestrate(task, context)`** — Simpler pipeline used by `/orchestrator/process`: orchestrator decides agents → each agent runs → consolidate recommendations.

### 3.4 Agent Prompts (all 8)

Each prompt instructs the agent to return `{"analysis": "...", "recommendations": [...], "confidence": 0.0-1.0}` JSON. The orchestrator prompt specifically returns `{"agents_involved": [...], "reasoning": "...", "urgency": "..."}`.

| DataFrame | Rows | Generator Function | Key Fields |
|---|---|---|---|
| `DEMAND_DATA` | 240 (10 products × 24 months) | `generate_demand_data()` | product_id, month, actual_demand, forecast_demand, forecast_error |
| `INVENTORY_DATA` | 40 (10 products × 4 warehouses) | `generate_inventory_data()` | current_stock, reorder_point, safety_stock, days_of_supply, status |

**Shipment delay logic:** probability = 1.0 - (on_time_delivery / 100); delay 1-10 days; reasons: Port congestion, Weather, Customs, Carrier capacity, Documentation, Equipment failure

## 5. ML Module — `backend/ml/`

| File | Purpose |
|---|---|
| `DataCoSupplyChainDataset.csv` | ~96 MB real-world supply chain CSV dataset from DataCo (training source) |
| `model-train-improved.ipynb` | Jupyter notebook documenting data preprocessing, feature engineering, model selection, cross-validation, and threshold tuning on the DataCo dataset |
| `pipeline.pkl` | Serialised scikit-learn pipeline (~790 KB) — loaded at runtime by `score_supplier_risk` tool |
| `threshold.pkl` | Saved optimal decision threshold (binary classification boundary) applied after `predict_proba()` |

### 5.1 ML Model Integration — Supplier Risk Scoring

The trained pipeline is the **sole inference engine** for the `score_supplier_risk` tool. The integration path is:

```
User triggers Supplier Risk (wf-6, /api/supplier-risk/analyze, or business query)
       │
       ▼
score_supplier_risk(@tool in langchain_agents.py)
       │
       ├─ 1. get_supplier_risk_data(supplier_id) → fetches raw supplier attributes
       │       (reliability_score, on_time_delivery, quality_rating,
       │        financial_health, risk_factors, shipment stats, news sentiment)
       │
       ├─ 2. Feature Engineering
       │       Constructs the same feature vector the model was trained on:
       │       [reliability_score, on_time_delivery, quality_rating_scaled,
       │        financial_health_encoded, negative_news_count,
       │        historical_delay_rate_pct, avg_delay_days]
       │
       ├─ 3. import joblib; pipeline = joblib.load("ml/pipeline.pkl")
       │       threshold = joblib.load("ml/threshold.pkl")
       │
       ├─ 4. prob = pipeline.predict_proba([features])[0][1]   # risk probability
       │       score = round(prob * 100, 1)                     # 0–100 scale
       │       label = "high" if prob ≥ threshold else "low"    # or critical/medium
       │
       └─ 5. Returns JSON:
               {risk_score, risk_level, component_scores, recommendation, ...}
               (identical schema to the old heuristic output — UI unchanged)
```

**Key design principle:** The ML model output is formatted identically to what the heuristic formula previously produced. This means every downstream system — the `business_query()` pipeline, `wf-6` workflow handler, `SupplierRiskDemo` page, message bus, and MongoDB logging — continues to work without modification. Only the calculation engine changed.

**Model characteristics (from training notebook):**
- Trained on DataCo real-world supply chain records (~96 MB CSV)
- Features: supplier reliability metrics, delivery history, quality ratings, financial health encoding, geopolitical news sentiment count, historical delay statistics
- Pipeline includes preprocessing (scaling, encoding) + classifier
- `threshold.pkl` stores the optimised decision boundary (tuned for F1/precision-recall tradeoff on imbalanced supplier risk classes)
- Outputs a probability score which is rescaled to 0–100 for display

## 13. How to Run
